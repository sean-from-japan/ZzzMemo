# qcatch 次フェーズ改善案の相談

## 現在の状態（2026-03-22）

FastAPI + Vanilla JS の Web アプリとして再実装済み。主要機能はすべて動作中。

### 実装済みの技術的特徴
- SSE Streaming チャット（Gemini Function Calling → 最終テキストのみストリーム）
- BroadcastChannel による複数タブ間同期
- バックグラウンドスレッド保存（threading.Lock + daemon Thread）
- 30日アーカイブ自動化
- インライン編集（タスクテキスト・カテゴリ・チェックリスト名/アイテム）
- サブタグ分割提案（カテゴリ5件以上でAIが具体的なグループ名を提案）
- チャット履歴永続化（chat_history.json）

### データモデル
```
Task: id, text, status(inbox/todo/done/trashed), category(5種), tags[], created_at, completed_at
ChecklistTemplate: id, name, items[], due_date
RecurringRule: id, text, category, frequency(daily/weekly/monthly), days_of_week, day_of_month
```

## 質問1: タスクの自動優先度スコアリング

現在、タスクに優先度フィールドはなく、カテゴリと作成日時でしか分類できません。

**検討中のアプローチ:**
- created_at からの経過日数（滞留ペナルティ）
- カテゴリ別の重み（仕事 > プライベート など、ユーザー設定可能）
- 期日（チェックリスト）との連携
- タスクテキストをAIが読んで緊急度を判定（都度API呼び出しは重い）

**質問:**
1. ローカルで動くルールベースの優先度スコアリングの設計案を提示してください。
2. 「AIが優先度を判定する」場合、全タスクを一括で送るバッチ処理か、サブタグ提案フローに組み込むかどちらが良いですか？
3. ダッシュボードの表示順をスコアベースにする場合、ユーザーが「なぜこの順番か」を理解できるようにするUIのヒントは？

## 質問2: PWA（オフライン対応）の最小実装

qcatch はローカルサーバー（localhost:5000）で動くため、インターネット不要です。
ただし「ブラウザのタブを閉じても使える」「ホーム画面に追加できる」を実現したい。

**制約:**
- バックエンドは Python (FastAPI) でローカル動作
- フロントエンドは単一 index.html（ビルドステップなし）
- Service Worker でのキャッシュが有効かどうかを確認したい

**質問:**
1. localhost の FastAPI に対して Service Worker を使うことはできますか？（セキュリティ制約は？）
2. 最小限の manifest.json と Service Worker で「ホーム画面追加」だけ実現するコードを教えてください。
3. FastAPI 側で manifest.json と Service Worker のファイルを配信する方法は？

## 質問3: チャットのプロアクティブ機能

現在のチャットは「ユーザーが話しかけたら応答する」のみです。

**検討中のアプローチ:**

- 滞留タスクが多いときだけアドバイスを表示
- 「朝のブリーフィング」ボタン（1クリックで今日の優先タスクを分析）

**質問:**
1. 「起動時自動メッセージ」はUXとして有効ですか？それとも煩わしいですか？
2. 「朝のブリーフィング」ボタンを実装する場合、プロンプトの設計はどうすべきですか？
   （例: 今日完了すべきタスクを3つ選んで理由を述べよ、など）
